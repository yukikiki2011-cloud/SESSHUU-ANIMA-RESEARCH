# slide-builder スキル

## 基本フロー

標準動作は、いきなりHTMLを生成せず、brief を揃えてから構成案を提示し、承認後にスライド生成へ進む。

1. brief を確認する
2. ページ割と各ページの1メッセージを構成案として提示する
3. ユーザーの承認または修正指示を受ける
4. 承認後に HTML/CSS/JS を生成する
5. validator と目視で品質チェックする

**例外**:
- ユーザーが `確認不要`、`すぐ生成`、`構成確認なし`、`ラフでよい` と明示した場合のみ、構成承認フェーズを省略して即生成してよい
- ユーザーが構成案だけを求めた場合は、構成提示で止める

## スライドレイアウト型（F1〜F8）

planning / seminar-architect からスライド型の指定がある場合、これを各スライドのレイアウト設計の起点にする。

| # | 型名 | 構成 | 効く場面 |
|---|------|------|----------|
| F1 | **ステートメント** | 1文だけ大きく。余白で強調 | コアメッセージ、セクション切替、問いかけ |
| F2 | **図解+テキスト** | 左右or上下に分割。片側に図、片側に説明 | プロセス説明、構造の視覚化 |
| F3 | **対比** | 2つの状態を並べる | Before/After、旧vs新、A vs B |
| F4 | **データビジュアル** | 数字・グラフ・チャートが主役 | 統計、成果、市場データ |
| F5 | **イメージフル** | 写真・イラストが画面の80%以上 | 感情喚起、事例のリアリティ |
| F6 | **ステップ** | 番号付きの手順を段階的に表示 | How-to、ワークの手順 |
| F7 | **引用** | 発言や一文を大きく。出典を小さく | 権威引用、参加者の声、衝撃的な事実 |
| F8 | **ブリッジ** | アクセントカラー全面+白文字で短いメッセージ | セクション切替、空気の転換 |

## 起動時インテーク

スキル発動直後、まず brief の不足を埋める。テーマだけで始めてもよいが、配布スキルとしては下記を揃えた方が品質が安定する。

### brief の必須項目
- テーマ
- 対象者
- 目的
- 持ち帰り

### brief の推奨項目
- 使う場面（提案 / 研修 / 採用 / 社内共有 / ウェビナー 等）
- ページ数
- 文字量
- テイスト
- 素材の有無（原稿あり / 箇条書きあり / おまかせ）
- 出力形式

### 情報が足りない場合 → インテーク（1メッセージで一括質問）

```
スライド生成前に brief を確認します。未定は「おまかせ」でOKです。

1. **スライドの使い方**: 登壇・プレゼン用（声で補足する） / 配布・自己完結型（読んで伝わる） / おまかせ
2. **テーマ**:
3. **対象者**:
4. **目的**:
5. **持ち帰り**:
6. **使う場面**: 提案 / 研修 / 採用 / 社内共有 / ウェビナー / おまかせ
7. **ページ数**: ショート（5〜15枚）/ 標準（15〜30枚）/ 詳細（30枚〜）/ おまかせ
8. **テイスト**: シンプル / スマート / カラフル / ブランド指定（_____）/ おまかせ
9. **素材**: 原稿あり / 箇条書きあり / おまかせ
10. **保存場所**: （例: ~/Desktop/my-slide）未指定の場合は必ず確認する
11. **進め方**: 構成を確認してから生成（標準） / すぐ生成
```

**デフォルト**:
- 明示がなければ `構成を確認してから生成` を選ぶ
- `すぐ生成` が明示されたときだけ承認フェーズを飛ばす
- 保存場所が未指定の場合は必ずインテークで確認する。デフォルトは `~/Desktop/<スライド名>` を提案して確認する

**インテークを簡略化してよい条件**:
- 依頼に `テーマ / 対象者 / 目的 / 持ち帰り` の4点が含まれている場合
- 足りない項目は 2〜4 個に圧縮して確認してよい

## 役割
スライド制作の専門家。planningの企画 or seminar-architectの構成 or 原稿を受け取り、2つのモードで出力する。

> planning / seminar-architect からスライド型（F1〜F8）の指定がある場合、それを各スライドのレイアウト設計の起点にする。

## モード

### 構成承認ドラフト（標準で必須）

HTMLモードで生成する前に、まずページ割とページごとの内容をユーザーへ提示する。承認が出るまでは HTML/CSS/JS を生成しない。

### 構成承認ドラフトの出し方
- 全体の狙いを 2〜4 行で要約する
- 全体枚数とセクション構成を先に示す
- 各ページは `1ページ = 1メッセージ` で短く出す
- 各ページに推奨レイアウト型（F1〜F8）を付ける
- 必要なら「追加確認したい論点」を最後に 1〜3 件だけ添える

### 構成承認ドラフトのフォーマット
```text
構成案 v1
全12枚

1. 表紙
メッセージ: 生成AI研修で現場の仕事はここまで変わる
入れる要素: タイトル / 副題 / キービジュアル
レイアウト: F5

2. 問題提起
メッセージ: AI導入が進まない理由はツール不足ではなく業務設計不足
入れる要素: 現状課題 3点 / 失敗例
レイアウト: F2
```

### 承認時の進め方
- 構成案の最後に `この構成でスライド生成に進めてよいですか？` と確認する
- 修正指示が来たら、まず構成案だけを更新する
- `OK`, `承認`, `これで進めて` などの明示が出た後に HTML 生成へ進む
- 承認前にデザイン実装へ進まない

### すぐ生成モード
- ユーザーが `すぐ生成` を選んだ場合でも、内部ではページ割を決めてから実装する
- ただし構成案の提示と承認待ちは省略してよい

### 構成モード（テキストベースの構成案）
下記の仕様に従って、テキストベースの構成案を作る。

**仕様**:
- 16:9（1920×1080px）、余白80px
- 背景白(#FFFFFF) / 文字黒(#1A1A1A) / アクセント1色統一
- 1ページ＝1メッセージ、60文字以内、箇条書き最大3項目
- 30〜100枚、各ページに推定表示秒数付記
- 合計3,300〜3,600秒（55〜60分）

**出力（1ページずつ）**:
```
スライド XX / 全XX枚
【セクション】【推定表示秒数】
【1メッセージ】【テキスト情報】【視覚情報】
【レイアウト】【配色】【演出・印象】
```

**追加ルール**:
- ストーリーライン: 問い→共感→発見→納得→行動
- 5枚連続同レイアウト禁止
- **スライド切替時に黒画面を出さない**: `body`と`.slide-stage`のbackgroundは`var(--c-base)`（白）にする。`#000`にするとopacity遷移中に黒フラッシュが見える
- **簡素に見せない**: 各スライドに装飾アクセント（上部アクセントライン、カード枠線、見出しアンダーライン、色面、背景グラデーション等）を入れる。テキスト+白背景だけのスライドは「簡素」と感じられる
- セクション切替にブリッジスライド（アクセントカラー背景+白文字）
- CTA最低3ページ（なぜ今→具体的ステップ→最後のひと押し）

### HTMLモード（Webスライド出力）
HTML/CSS/JSで動くスライド風Webサイトを生成する。
詳細なタイポ、配色、レイアウト、single-fileルールは `references/slide-web-standards.md` を読む。
日本語コピーの短文化・改行・収まり判定は `references/slide-copy-fitting.md` を併読する。

**前提**:
- 標準フローでは、構成承認ドラフトの承認後にだけ HTML 実装へ進む
- 承認前は HTML/CSS/JS を書き始めない

**出力モード**:
- **multi-file mode（デフォルト）**: `slides/index.html`, `css/style.css`, `js/app.js`, `js/slides.js` に分ける
- **single-file mode（明示時のみ）**: ユーザーが `single html`、`1ファイル`、`コピペ用`、`単体配布` を求めた時だけ、CSS/JS内包の自己完結HTMLを1本で出す

**スライドの使い方による設計分岐**:

インテーク項目1「スライドの使い方」の回答に応じて、以下のパラメータを切り替える。

| パラメータ | 登壇・プレゼン用 | 配布・自己完結型 |
|---|---|---|
| 本文フォント（slide-body） | 標準スケール（キャンバス短辺の約3.3%cqw） | 縮小スケール（キャンバス短辺の約2.2%cqw） |
| 1スライドの最大文字数 | 60文字 | 180文字 |
| 箇条書き最大項目数 | 3項目 | 5項目 |
| 1行の最大文字数 | 20文字 | 30文字 |
| ビジュアル比率 | 60%以上 | 40%以上 |
| 補足テキスト（data-notes） | 話者スクリプト（150〜300文字） | スライド内に本文として組み込む |
| 設計哲学 | 視覚が主役。声で詳細を補う | テキストが主役。読んで完結する |

「おまかせ」の場合、使う場面から推定する。登壇・ウェビナー→登壇用。配布・提案書→配布型。判断できない場合は登壇用をデフォルトにする。

**技術仕様**:
- `<section class="slide ..." data-section="【セクションID】">`
- 文字サイズ: コンテナクエリ単位（cqw）のカスタムクラスのみ使用
  - `slide-h1`, `slide-h2`, `slide-h3`, `slide-title`, `slide-body`
- **Tailwindの固定テキストサイズ（text-xl等）はスライド内で使用禁止**
- agendaItemsのidとdata-sectionを完全一致
- 改行品質を検証したい要素には `data-copy-role="headline|body|label|quote|metric"` を付け、役割を明示する

**slides.js アーキテクチャ（必須）**:

`js/slides.js` は「1スライド = 1関数」で書く。全スライドを1つの文字列に連結しない。

```js
// js/slides.js
// ⚠️ ES modules（export/import）は file:// では動かない。window グローバルを使う

// SLIDE 01: 表紙 | F5 | cover
function slide01Cover() {
  return `<section class="slide" data-section="cover" data-notes="...">
    ...
  </section>`;
}

// SLIDE 02: 問題提起 | F2 | problem
function slide02Problem() {
  return `<section class="slide" data-section="problem" data-notes="...">
    ...
  </section>`;
}

window.slideFactories = [slide01Cover, slide02Problem, /* ... */];
window.agendaItems = [
  { id: 'cover', label: '表紙' },
  { id: 'problem', label: '問題提起' },
  // ...
];
```

`index.html` では slides.js → app.js の順で通常の `<script>` タグで読み込む（`type="module"` 禁止）:

```html
<script src="js/slides.js?v=1"></script>
<script src="js/app.js?v=1"></script>
```

`js/app.js` では `window.slideFactories` / `window.agendaItems` をグローバルとして参照する（import 文不要）:

```js
// js/app.js（抜粋）
// import 文は書かない。slides.js が先に読み込まれ window.slideFactories が使える

const stage = document.querySelector('.slide-stage');
const rendered = new Set();

function ensureRendered(index) {
  if (rendered.has(index) || index < 0 || index >= slideFactories.length) return;
  const fragment = document.createRange().createContextualFragment(slideFactories[index]());
  stage.appendChild(fragment);
  rendered.add(index);
}

function goTo(index) {
  // 当該スライド + 前後1枚を先読み
  [index - 1, index, index + 1].forEach(i => ensureRendered(i));
  // active切替・スクロール処理
}

// 初期化: 最初の3枚だけ即レンダリング
[0, 1, 2].forEach(i => ensureRendered(i));
goTo(0);
```

これにより30枚超でも初期DOMが軽くなり、スライド関数単位でコピー流用できる。

**サイドバー（必須ルール）**:
- **デフォルト非表示・オーバーレイ式**。`position: fixed; transform: translateX(-sidebar-width)` で初期状態は画面外に格納する
- ナビバーの☰ハンバーガーボタンを押したときだけ `open` クラスを付与してスライドイン表示
- バックドロップ（半透明オーバーレイ）クリックまたはESCキーで閉じる
- サイドバーを常時表示にするとスライドの有効幅が削られ、コンテンツ設計が破綻する。**絶対に常時表示にしない**

**コンテンツ縦センター配置（必須ルール）**:
- `.slide-content` には必ず `justify-content: center` を設定する。デフォルト `flex-start` にすると内容が少ないスライドで下部に空白が溜まり「余白が多い」と指摘される
- グリッド・ステップリスト・比較カラム等の main コンテナには `flex: 1` を付けない。`flex: 1` にすると親の高さいっぱいに引き伸ばされ、センタリングが効かなくなる
- コードブロック等「縦に伸びてほしい」要素だけ inline style で `flex: 1` を付与する（他はデフォルトの auto）
- `.slide.active` は `display: flex; flex-direction: column` にして `.slide-content` が `flex: 1` で高さを取得できるようにする

**出力**:
- デフォルトは省略なしのmulti-file HTML/CSS/JS一式
- single-file mode時のみ、HTML/CSS/JS内包の単一HTML

**図解カタログの使い方（スライド実装の起点）**:

各スライドのHTMLを書く前に、以下の順で判断する:

1. そのスライドのF型を確認する
2. **F2（図解+テキスト）/ F4（データビジュアル）/ F6（ステップ）の場合**: `references/slide-diagram-catalog.html` を開き、D01〜D30から最適パターンを選ぶ
3. 選んだパターンのHTMLをコピーし、コンテンツを差し替える（ゼロから書かない）
4. カタログにぴったりのパターンがない場合のみ、最近いパターンをベースに改変する

| F型 | 優先して見るカタログ範囲 |
|---|---|
| F2 フロー・プロセス | D01〜D08（フロー・ステップ・タイムライン） |
| F2 構造・関係 | D09〜D15（マトリクス・ベン図・階層・ハブ） |
| F4 データ | D16〜D22（棒グラフ・折れ線・円グラフ・スコアカード） |
| F6 ステップ | D01〜D06（横フロー・縦フロー・ステップ番号） |

**トークスクリプト（data-notes）生成基準**:

全スライドの `<section class="slide" data-notes="...">` に話者用トークスクリプトを書く。

| 項目 | 基準 |
|---|---|
| 文字数 | 1スライドあたり150〜300文字（60〜120秒分） |
| 文体 | 口語体（〜です、〜ます、〜ですよね） |
| 構成 | ①このスライドで言いたいこと1文 → ②補足・説明2〜3文 → ③次のスライドへの橋渡し1文 |
| ビジュアルへの言及 | 図解・グラフ・写真を「指す」言葉を入れる（「この図を見てください」「右のグラフのとおり」） |
| 禁止 | スライドのテキストをそのまま読み上げるだけの原稿 / 体言止めのメモ書き |

例:
```
data-notes="このスライドでお伝えしたいのは、AI研修が現場に定着しない理由は、ツールの問題ではなく業務設計の問題だということです。ご覧の図のとおり、多くの企業がツール導入で止まってしまい、実際の業務フローへの組み込みができていません。次のスライドでは、定着させた企業が何をやったのかを具体的に見ていきます。"
```

**コピー整形ルール**:
- 見出し・本文は最初に改行なしで起草し、その後で `references/slide-copy-fitting.md` の文字数予算に合わせて整形する
- 文字数予算を超えた場合は、**フォントを縮める前に** コピー短文化 → レイアウト変更 → 幅調整の順で処理する
- 手動改行（`\n`, `<br>`）は文節・意味の切れ目でのみ入れる。助詞、句読点、単位、括弧閉じを行頭に残さない
- left-right レイアウトの強調文・数値横の短文は 1行 8〜10文字を上限とし、長くなる場合は body ブロックに逃がす
- 最終表示で `LAYOUT`, `PHOTO`, `CATEGORY`, `SECTION 01` などの開発用プレースホルダーを残さない

**品質チェック（実装後に必ず実行）**:
- **テキストはみ出し検出**: bundled validator `scripts/validate-slide-deck.mjs` または同等のPlaywright手順で、要素のboundingRectがスライド領域を超えていないかプログラムで検証する。目視だけに頼らない
- **改行位置**: 日本語の助詞（は・が・を・に・で・と・の）が行頭に来ていないか、1文字だけの末尾行が出ていないか確認する。単語の途中で折り返さない
- **accent/strong テキストは短く**: left-rightレイアウトの右カラムはフォントサイズが大きいため、1行あたり8〜10文字以内に収める。長い文は折り返しで崩れる
- **スクリーンショット画像**: X投稿等のキャプチャはプロフィールアイコン・名前・認証バッジが見切れないようにクロップする
- **ブラウザキャッシュ対策**: JS/CSSを更新したらindex.htmlの参照パスにクエリパラメータ（?v=N）を付けてキャッシュを回避する
- **ネストされたクリック要素**: スライド内のボタンやリンクには`event.stopPropagation()`を付け、スライド送りの誤作動を防ぐ
- **サイドバーはデフォルト非表示か**: 常時表示になっていたら即修正。ハンバーガーボタン+オーバーレイ式にする
- **下部余白の偏り**: スライドごとにスクリーンショットを撮り、コンテンツが上寄りで下に白空間が余っていないか確認する。`justify-content: center` が効いていれば上下均等になる
- **並列カードの高さを揃える**: 2〜4カラムでカードを横並びにする場合、カード下端に段差を作らない。`align-items: stretch` とカード側 `height: 100%` を基本にし、視覚的な整列を崩さない
- **判定基準は相対値で書く**: サイズ・余白・マージンの品質基準に固定 `px` を使わない。短辺比率、文字倍率、コントラスト比、要素間の比率で記述し、媒体サイズが変わっても同じ判定ができるようにする
- **人物画像の品位を守る**: 人物ビジュアルは実写、または実写と見分けがつかない品質のAI画像を使う。大袈裟な表情、品のない演出、安っぽい煽りは避ける

**Fact-check**:
- スライド内の数字・引用・企業名・事例が企画書および原典と一致しているか確認
- 誤りがあれば即座にHTML/CSSを修正

**Visual Evaluation**（HTMLファイル出力後に必ず実行）:
- **ステップ1（即時・必須）**: ファイル出力直後にlocalhost起動 → `scripts/validate-slide-deck.mjs` を実行する。目視確認の前にこれを通す。後回し禁止
- **ステップ2**: Playwrightで全スライドをスクリーンショット撮影し、レイアウト崩れ・余白偏り・はみ出しを目視確認する
- **ステップ3**: Sprint Contractの合格基準に照らして自動検証（コントラスト、フォントサイズ、レイアウト崩れ、スライド切替の黒画面）
- 320px / 1920px の2幅でスクリーンショット比較
- FIXがあれば即修正→再スクリーンショットでPASS確認
- **検証画像の後片付け（必須）**: Playwrightで撮影したスクリーンショット（`screenshot-*.png`, `check-*.png`, `v5-*.png` 等）は、検証完了後に即座に `/tmp/` へ移動する。プロジェクトディレクトリに残さない

## 脱AIデザインゲート（共通原則: `../shared/anti-ai-design.md`）

スライド生成時に以下を自動チェックする。違反があれば修正してから出力:

- [ ] 各スライドにカテゴリラベルではなく**主張文**（言いたいことを1行で言い切る）があるか
- [ ] **ビジュアルファースト原則を守っているか**: 各スライドで「このメッセージを図解・グラフ・表・写真で伝えられないか？」を最初に検討したか。テキスト箇条書きはビジュアル化できなかった情報にのみ使う。デッキ全体でテキスト主体スライドが45%を超えていないか
- [ ] 図解が必要な場面で**アイコン＋カードで代替していないか**（フロー・マトリクス・ベン図等の構造的図解を使う）
- [ ] ダークトーン＋ネオンカラーを常態化していないか（基本はホワイトベース。ダーク面は表紙・セクション区切り・ブランド必須時のみ）
- [ ] **フチ文字（`-webkit-text-stroke` / `text-stroke`）を使っていないか**（素人感・ゲーム感が出るため全面禁止）
- [ ] **ドロップシャドウを使っていないか**（`box-shadow`は原則禁止。どうしても区切りが必要な場合は`border`で代替）
- [ ] **光彩・グロー効果を使っていないか**（`text-shadow`のぼかし・`filter: drop-shadow()`・glow系CSS禁止）
- [ ] **過剰な装飾がないか**（グラデーションボーダー・回転した帯・波形の背景・浮き出し風装飾禁止。スタイルファイルで明示許可されたもの以外は使わない）
- [ ] 英語ラベル見出し（FEATURES, PRICING等）を使っていないか
- [ ] 絵文字を使っていないか
- [ ] 仮置きプレースホルダー（LAYOUT, PHOTO, CATEGORY, COMPANY NAME, Lorem ipsum 等）が残っていないか
- [ ] 全スライド同じレイアウト・同じ余白になっていないか（5枚連続同レイアウト禁止は既存ルール）
- [ ] Web的なスカスカではなく、実務資料らしい情報密度があるか
